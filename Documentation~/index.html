Installation

1. Add the Keijiro scoped registry

Open Edit → Project Settings → Package Manager and add:

NameKeijiro
URLhttps://registry.npmjs.com
Scope(s)jp.keijiro

2. Install Minis and RtMidi

In Window → Package Manager → My Registries, install jp.keijiro.minis. It pulls in jp.keijiro.rtmidi as a dependency.

3. Install this package

In Package Manager → + → Add package from git URL:

https://github.com/caseyfarina/midiFighterForUnity.git

4. Update your Midi Fighter 64 firmware

Only firmware 24 Jul 2017 or later supports the full 128-color palette this package targets. Download the latest Midi Fighter Utility and click Update Firmware.

Utility settings required: Bank 1, MIDI Channel 3, MIDI Type = Notes, Corner Button Bank Change = disabled.

Quick start

Two paths:

A. Use the generated test scene

  1. Import the sample: Package Manager → Samples → Test Scene → Import
  2. Run Tools → MidiFighter64 → Create Test Scene
  3. Press Play — press pads on the MF64 to light spheres on-screen and LEDs on the hardware

B. Build your own

  1. Add a GameObject to your scene, attach MidiEventManager, UnityMainThreadDispatcher, MidiGridRouter (and MidiMixRouter if using MIDI Mix)
  2. Add MidiFighterButtonRouter if you want Button/Toggle semantics with LED feedback
  3. Add MidiFighterOutput to drive the LEDs
  4. In your own script, subscribe to the events you care about (see API reference)
using MidiFighter64;

void OnEnable()
{
    MidiFighterButtonRouter.OnButtonPress += HandlePress;
    MidiFighterButtonRouter.OnToggle      += HandleToggle;
    MidiMixRouter.OnKnob                  += HandleKnob;
}

void OnDisable()
{
    MidiFighterButtonRouter.OnButtonPress -= HandlePress;
    MidiFighterButtonRouter.OnToggle      -= HandleToggle;
    MidiMixRouter.OnKnob                  -= HandleKnob;
}

void HandlePress(GridButton btn, float velocity) { /* ... */ }
void HandleToggle(GridButton btn, bool isOn)     { /* ... */ }
void HandleKnob(int channel, int row, float val) { /* ... */ }
Always unsubscribe in OnDisable. Router events are static — a subscribed component that's destroyed without unsubscribing will keep the object alive and fire on stale state.

Button / Toggle modes

MidiFighterButtonRouter gives each pad one of two behaviors:

  • Button — fires OnButtonPress on press, OnButtonHold each frame while held, OnButtonRelease on release
  • Toggle — fires OnToggle(btn, true/false) on press; ignores release

Default is Button for every pad. Assign a MidiFighter64ButtonConfig ScriptableObject to the router's Config slot to make specific pads toggle.

Create a Button Config

  1. Project window → right click → Create → MidiFighter64 → Button Config
  2. In the Inspector's 8×8 grid, tick the pads you want as toggles
  3. Drag the asset onto the router's Config field

LED output

MidiFighterOutput writes to the MF64's LEDs via jp.keijiro.rtmidi — cross-platform (Windows / macOS / Linux). Drop the component on any GameObject; it auto-discovers the device, hot-plug re-opens on port changes.

using MidiFighter64;

// Raw velocity (0–127):
MidiFighterOutput.Instance.SetLED(note: 64, velocity: 3);

// Or via the palette enum:
MidiFighterOutput.Instance.SetLED(64, MidiFighterLEDColor.White);

MidiFighterOutput.Instance.ClearLED(64);
MidiFighterOutput.Instance.ClearAllLEDs();
MidiFighterOutput.Instance.SetAllLEDs((int)MidiFighterLEDColor.Blue);

// Animation channel (Ch4):
MidiFighterOutput.Instance.SetAnimation(64, velocity: 20);

The MidiFighterButtonRouter can drive LEDs automatically when Drive Toggle LEDs or Drive Button LEDs is checked in the Inspector. You pick the on/off colors from a dropdown.

Clear-on-Start: MidiFighterOutput clears all 64 LEDs at scene start and scene exit by default so the hardware doesn't carry stale state between Play sessions. Uncheck Clear On Start to keep whatever state was there.

Akai MIDI Mix

MidiMixRouter exposes the mixer's controls as typed events. Add it to your scene; subscribe:

using MidiFighter64;

MidiMixRouter.OnKnob           += (ch, row, v) => { /* ch 1-8, row 1-3, v 0-1 */ };
MidiMixRouter.OnChannelFader   += (ch, v)     => { /* ch 1-8, v 0-1 */ };
MidiMixRouter.OnMasterFader    += v            => { /* v 0-1 */ };
MidiMixRouter.OnMute           += (ch, on)    => { /* fires when SOLO is NOT held */ };
MidiMixRouter.OnSolo           += (ch, on)    => { /* Mute button pressed while SOLO held */ };
MidiMixRouter.OnRecArm         += (ch, on)    => { };
MidiMixRouter.OnSoloModifier   += isDown       => { /* SOLO modifier button state */ };
MidiMixRouter.OnBankLeft       += ()           => { };
MidiMixRouter.OnBankRight      += ()           => { };

// Or query the modifier state on demand:
if (MidiMixRouter.IsSoloHeld) { /* ... */ }
Solo behavior: the mixer's SOLO button is a modifier. It emits its own note (27) on press and release, tracked via OnSoloModifier(bool) and IsSoloHeld. While it's held, the 8 Mute buttons emit the Solo note set and fire OnSolo instead of OnMute. Rec Arm buttons are not affected.
Send All button: pressing it causes the hardware to re-emit all 33 CC messages (24 knobs + 8 channel faders + master fader). No dedicated note is sent — the router just sees a burst of CC events.

Latching Mute / Rec-Arm

The Mute and Rec-Arm buttons are momentary in hardware, but the router latches them by default: LatchMute and LatchRecArm are both true. Press once to turn on, press again to turn off.

// With latching on (the default), OnMute fires once per press and carries
// the NEW latched state. Note-off is ignored.
MidiMixRouter.OnMute += (ch, isOn) => { /* isOn = latched state */ };

// Untick LatchMute / LatchRecArm for momentary behavior, where the events
// fire on both note-on and note-off with the raw button state.

MidiMixOutput mirrors these router events rather than raw MIDI, so the hardware LED holds for as long as the button is latched on, with no code from you. The status drawer holds its pad filled to match.

OnSolo is always momentary. The SOLO modifier has to be held for those notes to be emitted at all, so there is no latch to hold.

Filtering MIDI devices

MidiEventManager connects to every MIDI input port on the system and merges them into one event stream. That is usually what you want — but if a MIDI monitor, a loopback port (loopMIDI), or a network session is carrying a copy of your controller's traffic, every message arrives twice.

Duplicate delivery is invisible to momentary handlers (on/on then off/off lands in the same place) but it silently breaks anything press-to-toggle: the latch flips on and straight back off within one frame, so the button looks dead while the raw MIDI looks perfect. MidiEventManager warns once per session when it sees the same note arrive from two ports in the same frame.

// Substring match against the port name, case-insensitive.
MidiEventManager.Instance.SetDeviceFilter(
    allowed: new[] { "Fighter", "MIDI Mix" },   // empty = accept every port
    blocked: new[] { "loopMIDI" });             // applied after the allow list

Both lists are fields on MidiEventManager. The shipped MIDI Controller prefab sets the allow list to the two controllers this package targets; the component's own default is empty, meaning every port is accepted. Skipped ports are logged.

MIDI Mix LED echo

The Mute (amber) and Rec-Arm (red) buttons on the MIDI Mix do not self-illuminate on press. The hardware only lights them when it receives a matching Note On back from the host. Add MidiMixOutput to your scene and it will auto-mirror presses.

using MidiFighter64;

// Automatic (default): LEDs light on press, clear on release. No code needed.

// Manual — for latched toggle behavior:
MidiMixOutput.Instance.SetMuteLED(channel: 1, lit: true);
MidiMixOutput.Instance.SetRecArmLED(channel: 3, lit: true);
MidiMixOutput.Instance.SetSoloLED(channel: 5, lit: true);   // Mute-while-SOLO-held
MidiMixOutput.Instance.ClearAllLEDs();

For fully custom state, uncheck Auto Mirror Buttons on MidiMixOutput, subscribe to MidiMixRouter.OnMute, and call SetMuteLED yourself. You don't need to do this just to get toggle behavior — see latching above.

Status drawer overlay

MidiStatusDrawer (Test Scene sample) is a screen-space overlay that mirrors both controllers live: the 8×8 pad grid, all 24 knobs, 9 faders, the mute / rec-arm / solo / bank buttons, and a readout of the most recent MIDI event. It renders on every active display and is read-only — all widgets ignore picking, so clicks pass straight through to your scene.

It ships on the MIDI Controller prefab. Delete the component from your scene instance to drop the overlay, or add it to any GameObject to use it standalone.

Widgets render at 40% opacity until they receive their first MIDI event, then snap to full — which distinguishes "not yet touched" from "value is zero". Theme, opacity and stroke weight all restyle in place, so changing them never resets that state.

Test scene

The Test Scene sample gives you a running visual + LED integration test in one click.

  1. Package Manager → Midi Fighter 64 → Samples → Test Scene → Import
  2. Tools → MidiFighter64 → Create Test Scene — saves MidiControllersTestScene.unity under Assets/
  3. Press Play

The scene includes:

  • An 8×8 sphere grid mapped 1:1 to MF64 pads. Pressing a pad turns the matching sphere the router's Button Down color.
  • A Play Wave UI button that animates a left-to-right sweep on-screen and on the physical LEDs.
  • The status drawer overlay — press ` to show it.
  • A raw MIDI event log overlay for debugging.

Drag Runtime/MIDI Controller.prefab into a scene and everything works. It carries all eight components — MIDI input, both routers, both LED outputs, and the status drawer — already wired and configured. Every setting lives on the component that owns it: device filtering on MidiEventManager, the MF64 pad mode / color grid and LED colors on MidiFighterButtonRouter, latching on MidiMixRouter, and the drawer settings above on MidiStatusDrawer. Select the prefab instance in your scene to configure it.

Troubleshooting

No MIDI events fire

  • Is MidiEventManager in the scene? Nothing works without it.
  • The first event on any channel is sometimes swallowed by Minis — press a pad twice.
  • Try disconnecting via a USB hub. Some hubs cause detection issues.

Mute / Rec-Arm toggles don't hold — the light flashes and goes out

  • Almost always duplicate MIDI delivery. A second port is carrying a copy of your controller's traffic, so each press is handled twice: the latch flips on and straight back off in the same frame. Check the console for the "arrived from two MIDI ports in the same frame" warning, which names both ports.
  • Culprits are MIDI monitors, loopback ports (loopMIDI), and network MIDI sessions. Fix it with device filtering rather than by closing the tool.
  • Note that momentary mode looks perfectly fine under the same fault, so "the buttons work until I enable latching" points here, not at the latch code.

Wrong colors on LEDs

  • Your firmware is pre-24 Jul 2017. Update via the Midi Fighter Utility.
  • Utility's Base Channel must be 3. MIDI Type must be Notes.

Spheres render magenta in URP

  • Fixed as of v1.1.0 — MidiFighterTestScene now reuses the material Unity's CreatePrimitive assigns, which has the correct pipeline shader.

LEDs stay lit after exiting Play

  • The MF64 remembers state until it receives clear messages. Enable Clear On Start on MidiFighterOutput.

Compile error: RtMidi namespace not found

  • jp.keijiro.rtmidi isn't installed. Add the scoped registry, then install Minis — it pulls RtMidi in.
